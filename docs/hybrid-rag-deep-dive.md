# 从一次客服对话看 RAG 的演进：当"我的订单还没到"变成一道技术题

## 引子

假设你是一家电商平台的客服系统负责人。一个叫小明的用户发来一条消息：

> "我两周前下的订单还没到，帮我查一下怎么回事"

这条看起来简单的消息，背后涉及的问题比你想象的多得多：

- "两周前" —— 需要理解时间表达
- "订单" —— 是查物流、查退款、还是查商品？
- "还没到" —— 送货延迟还是根本就没发货？
- 如果小明接着问 "退货运费谁出"，这是同一个会话还是一个新的问题？

下面我们跟着这条消息，走一遍 RAG 系统的演化之路。

---

## 第一章：关键词匹配时代

### 遇到的第一问题：查不到

最朴素的做法，把用户问题拆成关键词，去知识库里匹配：

```
用户： "我的订单两周了还没到"
分词： [订单, 两周, 没到]
匹配： 遍历知识库文档，找包含这些词的
```

你查了下，知识库里有这么几条：

```
文档1: "Shipping Policy — Standard shipping takes 3-5 business days..."
文档2: "Return Policy — Items can be returned within 30 days of delivery..."
文档3: "Order Tracking — You can track your order from the Orders page..."
```

**结果：一条都没命中的。**

为什么？因为你的知识库是英文的，用户说的是中文。"订单"确实碰巧匹配 title 里的 "Order"，但 "没到" 匹配不了任何内容。更隐蔽的问题是 —— 就算你改成中文知识库，"两周" 和 "3-5 business days" 也匹配不上，因为它们是数值表达而非关键词。

**这是关键词搜索的第一个死穴：语言不通、表达方式多变，关键词稍微一变就漏。**

### 然后发现的问题：查不准

后来你把知识库补充了中文版本：

```
文档1: 发货政策 — 标准发货 3-5 个工作日，急件 1-2 个工作日
文档2: 退货政策 — 收货后 30 天内可退货
文档3: 订单查询 — 可在订单页查看物流信息
```

小明说 "订单" → 命中文档 1、2、3，全中。但是：

小明实际想问的是 **"为什么两周了还没到"**，属于物流问题。你希望能优先返回文档 3（订单查询），但关键词匹配不分主次，只要包含就算命中。当知识库扩展到几百条文档后，"订单" 这个词出现在一半的文档里，**召回率 100%，准确率不到 10%**。

更麻烦的场景：用户问 "退货运费谁出"——"退货" 命中文档 2（退货政策），"运费" 命中文档 1（发货政策）。两条文档的关键词权重一模一样，无法判断用户到底想问退货还是发货。

**关键词搜索的第二个死穴：无法区分"包含"和"相关"之间的区别。**

---

## 第二章：向量检索时代

### 语义理解解决了语言不通

你引入了 Embedding 模型，把所有文档转成向量，用户的查询也转成向量，算余弦相似度。

现在小明说 "我的订单两周了还没到"，向量检索的结果：

```
文档3 (订单查询):  相似度 0.91
文档1 (发货政策):  相似度 0.74
文档2 (退货政策):  相似度 0.32
```

**语义搜索正确地把 "还没到" 和物流/发货关联起来了。** 语言不再是障碍，中文查中文文档自然没问题，甚至用户说 "where is my order" 也能正确匹配。

更有意思的是，现在用户说 "怎么退钱" 也能匹配到退货政策，而之前的关键词搜索会因为 "退钱" 不等于 "退货" 而漏掉。

**这是向量检索给客服系统带来的第一个质变：能理解"意思"了。**

### 新的问题来了：精确匹配退化

有一天小明问：

> "我的优惠券代码 VIP50 怎么用不了"

向量检索的结果：

```
文档1: 优惠券政策 — 相似度 0.85
文档4: 支付方式 — 相似度 0.72
文档5: 账户管理 — 相似度 0.68
```

看起来不错。但如果你仔细看，文档 1 里根本没有 "VIP50" 这个具体的券码。向量检索找到的是"优惠券"这个概念，而不是 "VIP50" 这个实体。

**向量检索的通病：不识字，只看"意思"。**

这让客服系统在处理订单号、券码、手机号、地址这些精确信息时特别吃亏。用户报了一个订单号 "ORD-987654"，向量检索可能把它跟所有带 "order/订单" 字眼的文档匹配上，却无法精确对应到该订单的记录。

**更隐蔽的问题：领域偏移。**

你的 Embedding 模型是通用模型，在电商场景下效果还行。但如果你的客服系统是医疗或法律领域的：

> "我父亲体检发现 CA19-9 指标偏高，需要进一步检查吗"

通用 Embedding 可能会把 "CA19-9" 理解为数学或者化学相关的内容，而不是肿瘤标志物。**模型没见过的领域知识，向量化之后会严重变形。**

这就是为什么 Elastic 在他们的生产环境复盘里说了一句很有名的话：

> **"Partial retrieval is worse than no retrieval."**

当你检索到一堆"看起来相关但实际不相关"的文档喂给 LLM 时，LLM 会被这些干扰信息带偏，回答质量比完全不检索还要差。Elastic 的数据是：低质量上下文评分 8.15/10，无上下文反而 9.18/10。

## 第三章：混合检索时代

### 让精确和语义同时工作

你意识到不能只依赖一种检索方式，于是做了双通道：

```
小明说 "订单 ORD-987654 两周没到"
          │
    ┌─────┴─────┐
    │           │
  BM25       向量
(关键词)    (语义)
    │           │
  ORD-987654   订单跟踪
  精确匹配     语义匹配
    │           │
    └─────┬─────┘
          │
      RRF 融合
```

BM25 通道做了什么？它把 "ORD-987654" 作为一个精确词，去文档全文里匹配。任何包含 "ORD-987654" 的文档（比如订单记录）会拿到极高的分数。

向量通道做了什么？它把 "订单两周没到" 做语义匹配，找到了发货政策和订单查询文档。

**问题来了：BM25 的分和向量的分不在一个量级，怎么合并？**

BM25 的分值范围取决于文档集合大小和词频分布，可能是 0~15。向量的余弦相似度是 0~1。你不能说 "BM25 10 分 + 向量 0.8 分 = 10.8"，因为这样 BM25 完全主导了结果。

这就是 **RRF（ Reciprocal Rank Fusion）** 登场的地方。

### RRF 的原理：不看分数，看排名

RRF 不关心你得了多少分，只关心你排第几：

```
RRF 分 = 1 / (60 + 排名)

排名第 1  → 1/61 ≈ 0.0164
排名第 5  → 1/65 ≈ 0.0154
排名第 10 → 1/70 ≈ 0.0143
```

在 BM25 里排第 1 和在向量里排第 1，得到的 RRF 贡献是一样的。**这就解决了两种分数不可比的问题。**

回到小明的场景：

```
文档 A (订单 ORD-987654 记录):     BM25 第 1 → RRF 0.0164  向量未命中 → RRF 0    合计 0.0164
文档 B (订单查询帮助):              BM25 第 5 → RRF 0.0154  向量第 1 → RRF 0.0164  合计 0.0318
文档 C (发货政策):                  BM25 第 8 → RRF 0.0147  向量第 3 → RRF 0.0159  合计 0.0306
```

最终排序是：文档 B > 文档 C > 文档 A。

**BM25 贡献了精确匹配（ORD-987654），向量贡献了语义匹配（订单查询），RRF 让两者各自加分。** 小明既得到了订单相关的帮助文档，同时也关联到了他具体的订单记录。而在纯向量时代，"ORD-987654" 这个信息基本是浪费的。

### 但还不够：当两个渠道都犯错

假设小明的知识库里有一条文档讲的是 "员工年度体检安排"，因为包含了 "体检" 两个字，被 BM25 匹配上了（但实际跟客服问题无关）。同时向量检索也因为 "体检" 这个词的语义关系把它排进了前 10。

RRF 融合后，这条无关文档排到了第 4 位。**RRF 不会判断相关性——它只负责排序合并。**

引入 Reranker（重排序器）来解决：

```
BM25 + 向量 RRF 融合 → 取 Top-20
  → 用 Cross-encoder 对这 20 条逐个打分（0~1 相关性）
  → 重新排序取 Top-3
  → 喂给 LLM
```

Cross-encoder 把 query 和 doc **成对**输入模型，输出一个相关性分数。它比 Embedding 双向量的点积准确得多，但计算量也大得多——所以只对 RRF 之后的 Top-20 做精排，不做全量检索。

加上 Reranker 之后，那条"员工体检"的文档被准确判定为不相关，排到了 20 名之后。

**最终的检索管线就变成了：**

```
用户查询
    │
├─ BM25（精确匹配）──┐
├─ 向量（语义匹配）──┼─ RRF 融合 → Reranker 精排 → Top-3 → LLM
                    ┘
```

## 第四章：Agentic RAG 时代

### 前面的检索仍然有一个根本问题

小明说 "我的订单还没到"，检索找到了物流相关的文档，LLM 给出了标准回复：

> "您的订单预计 3-5 个工作日送达，请耐心等待。"

但这个回答没有任何用处——**因为系统不知道小明到底订的是什么、什么时候下的单、现在的物流状态是什么。**

检索只能返回"静态知识"。但客服场景中，大量问题需要"动态操作"：

- 查订单状态 → 需要调订单系统 API
- 修改订单地址 → 需要调 CRM 系统
- 创建退款工单 → 需要调售后系统
- 查优惠券 → 需要调营销系统

### Agent 的介入：从"搜到什么回什么"到"决定怎么做"

现在的架构变成了这样：

```
用户: "我的订单两周了还没到"
                    │
               Supervisor Agent
                    │
         ┌──────────┼──────────┐
         │          │          │
    intent_    knowledge_   ticket_
    classify   rag         handler
         │          │          │
    "订单查询"  发货政策   查 ORD-987654
                          实际物流状态
         │          │          │
         └──────────┼──────────┘
                    │
              LLM 综合回复
                    │
              "您的订单 ORD-987654
               已延迟，当前在转运中心。
               需要帮您催单吗？"
```

流程变成了：

1. **intent_classify** 判断意图为 "订单查询"
2. **knowledge_rag** 检索发货政策，给出标准信息
3. **ticket_handler** 直接调订单系统，查到 ORD-987654 的实际物流状态——"已延迟，目前在广州转运中心"
4. **LLM 综合**两条渠道的信息，给出真正有用的回答

小明接着问：

> "退货运费谁出"

这不是一个问题了——这是一次**意图切换**。Agent 的能力就在于它能感知上下文的变化：

1. **intent_classify** 判断新意图为 "退货咨询"
2. 上下文里已经有订单 ORD-987654，自动带入
3. **knowledge_rag** 检索退货政策
4. 结合订单信息判断：ORD-987654 收货超过 30 天了吗？没有 → 可退货。运费？用户原因退货运费自付。
5. **compliance_check** 检查回复里有没有泄露用户隐私信息

Agent 可以自主决定：
- 什么时候该查知识库
- 什么时候该调操作接口
- 什么时候该问用户更多信息
- 什么时候该承认不知道，建议转人工

### 置信度的作用

回到那句话："Partial retrieval is worse than no retrieval."

Agentic RAG 的关键设计之一是**置信度阈值**。当知识_rag 检索到的文档平均相关性低于阈值（比如 0.6），Agent 不会强行凑答案，而是：

> "抱歉，我没能在知识库中找到相关信息。让我帮您转接人工客服。"

这不是失败——这是保护用户体验。Elastic 的数据表明，用户对"我不知道"的满意度，远高于对"错误回答"的满意度。

## 总结：一条消息走过的路

回顾小明的这条消息：

> "我的订单两周了还没到，帮我查一下"

| 阶段 | 结果 | 问题 |
|------|------|------|
| 纯关键词 | 匹配失败，回复"没有相关信息" | 语言不通、表达方式不灵活 |
| 纯向量 | 找到发货政策，回复"预计 3-5 天" | 语义正确但无法处理订单号，回答无实际价值 |
| BM25 + 向量 + RRF | 同时匹配订单记录和发货政策 | 排序合理但无法做实际操作 |
| + Reranker | 过滤掉无关干扰项 | 检索质量更精准 |
| Agentic RAG | 先分类意图，再查文档，再查系统，综合回复 | 能查询、能操作、能感知上下文切换 |

**现在回头看 Elastic 那句名言，就会发现它的真正含义：**

不是"不要做检索"，而是**不要只做一次检索就草草回答**。检索只是第一步，后续的融合、精排、置信度判断、Agent 决策，每一步都在决定用户体验的最终质量。
